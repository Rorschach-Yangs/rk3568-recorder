# 分布式光纤声波感知边缘计算终端

## —— 基于 RK3568 + φ-OTDR 的工业级数据记录与智能诊断系统

---

## 1. 项目概览

本项目设计了一台**集高速同步采集、大容量固态存储、边缘AI诊断于一体**的工业级嵌入式终端。它以 Rockchip RK3568 为主控，结合相干光时域反射（φ-OTDR）技术，实现对长距离光纤沿线振动/应变/温度事件的实时监测、精准定位和智能分类。

**一句话说清楚**：把激光干涉产生的微弱相位信号变成 8 通道高速数据流，在本地完成采集、存储、分析，只上传诊断结果。

## 3. 物理层：从激光到电信号

### 3.1 φ-OTDR 原理

φ-OTDR（相位敏感光时域反射仪）利用窄线宽激光脉冲注入传感光纤，探测光纤中瑞利背向散射光的相位变化。当光纤某处受到振动或应变时，该处折射率变化导致背向散射光的相位发生偏移。

- **相干探测**：背向散射光与本地参考光在 BPD 上干涉，将纳米级的相位变化转换为可测量的 RF 电信号
- **空间定位**：激光脉冲往返时间映射为光纤距离：`d = c·Δt / (2n)`
- **多通道分集**：一根光纤上沿长度分为多个感知区间，各自对应一个采集通道

### 3.2 低噪声 AOM 驱动信号源

声光调制器（AOM）需要频率精确、相位噪声极低的 RF 驱动信号来产生稳定的移频。信号源的相位噪声直接限制了系统的应变分辨率。

| 指标 | 要求 | 原因 |
|------|------|------|
| 频率稳定度 | ±0.1 Hz @200MHz | 保证相干探测的拍频稳定 |
| 相位噪声 | <-120 dBc/Hz @1kHz offset | 噪声过高会湮没微弱相位信号 |
| 谐波抑制 | >50 dBc | 避免谐波产生伪拍频干扰 |

### 3.3 BPD 平衡探测

平衡探测器（Balanced Photo-Detector）接收两路光（信号光 + 参考光），差分输出消除共模强度噪声。BPD 的共模抑制比（CMRR）直接决定系统对激光器 RIN 噪声的免疫能力。

```
信号光 I_s ─→ PD1 ─→ │
                     ├─→ RF差分输出 (承载相位信息)
参考光 I_r ─→ PD2 ─→ │
```

### 4.3 边缘 AI 诊断引擎

AI 引擎每 100ms 巡检一次最新数据块，提取特征并分类事件：

| 事件类型 | 特征指纹 | 对应场景 |
|---------|---------|---------|
| 机械开挖 | 宽频高能量 + 冲击脉冲 | 挖掘机/镐头在光缆上方施工 |
| 锚击/锚落 | 瞬态尖峰 + 低频振荡 | 船只抛锚砸中海底光缆 |
| 电缆应变 | 低频持续上升 + RMS 渐变 | 电力电缆热胀冷缩/地质沉降 |
| 相位截断 | RMS 骤降至零 | 光纤断裂/连接器脱落 |
| 谐波畸变 | 高 THD + 多倍频分量 | 电机/变频器电磁干扰耦合 |
| 环境噪声 | RMS 在基线范围内波动 | 风雨/交通/正常工况 |

诊断结果仅上传事件类型、时间戳、严重程度、空间位置（由 PTP 时间戳换算），**带宽占用降低 90% 以上**。

## 5. PTP 授时与空间定位

### 5.1 为什么需要 PTP

在分布式 φ-OTDR 监测中，多个采集终端沿光纤部署在不同位置。PTP（IEEE 1588v2）通过以太网实现终端间**微秒级时钟同步**。

```
主时钟 (GPS/北斗) ─→ 交换机 ─→ RK3568终端A
                              └→ RK3568终端B
                              └→ RK3568终端C
```

### 5.2 从时间戳到空间位置

φ-OTDR 中，激光脉冲在光纤中的往返时间直接对应物理距离：

```
d = (c / 2n) × Δt

c = 3×10⁸ m/s (真空中光速)
n = 1.468 (光纤折射率)
Δt = 往返时间差

PTP精度 1μs → 定位精度 ≈ 102m
PTP精度 100ns → 定位精度 ≈ 10m
```

## 6. 设备树 (DTS) 与系统定制

RK3568 的硬件引脚配置通过设备树（Device Tree Source）告知 Linux 内核。这是一个描述硬件接线关系的结构化文本文件，编译为 `.dtb` 二进制后在内核启动时加载。

**需要配置的关键节点**：

- AD7606 的 SPI 片选、BUSY 中断引脚、CONVST 触发引脚、DMA 通道
- SATA 控制器的 PHY 参数和电源策略
- 以太网 MAC 的 PTP 硬件时间戳 (PTP Hardware Clock) 支持
- GPIO 看门狗复位引脚

系统层面需要开启 `PREEMPT_RT` 实时内核补丁、SATA UAS 驱动、SPI DMA 引擎、PTP 1588 协议栈。

## 9. 面试追问应对

### Q1: "这个项目最难的地方是什么？"

> "最难的是 ADC 采集的 DMA 驱动。200kSPS 意味着每 5 微秒要完成一次 8 路同步采集，如果用中断模式每秒 20 万次中断 CPU 直接打满。必须配置 SPI DMA 控制器把采样数据直接搬运到环形缓冲区的物理地址，中间不能经过 CPU memcpy。这需要在 DTS 里正确配置 DMA 通道、SPI 片选、BUSY 引脚中断映射，还要处理 DMA 半满/全满中断来驱动乒乓缓冲。"

### Q2: "AI 模型怎么部署到 NPU 上的？"

> "分两步。第一步在 x86 开发机上用 RKNN-Toolkit2 把训练好的 PyTorch 模型转成 INT8 量化的 .rknn 格式，同时做精度验证确保量化损失在可接受范围。第二步在 RK3568 上链接 librknnrt.so，用四个 C API 完成推理：rknn_init 加载模型、rknn_inputs_set 喂入特征、rknn_run 触发 NPU 运算、rknn_outputs_get 拿分类结果。整个过程不到 50 行 C 代码。"

### Q3: "掉电了数据怎么办？"

> "三个层面的保障。软件层用 fdatasync 确保数据从内核页缓存刷到 SSD。文件系统层用 ext4 journal 模式保证元数据一致性。硬件层在供电设计上预留了超级电容，掉电瞬间给 SSD 提供 30ms 续流，足够把 SSD 内部 DRAM 缓存中的数据写回 NAND。"

### Q4: "为什么不做云边协同，全部本地处理？"

> "带宽是硬约束。8 通道 200kSPS 产生 3.2 MB/s 原始数据，一天就是 276 GB。如果我们监控的是海底电缆，卫星回传带宽可能只有几 Mbps，根本传不回去。所以必须在边缘做 AI 抽稀，只上传事件摘要。但设备同时保留了原始波形的本地存储，可以用来事后追溯和模型迭代。"
